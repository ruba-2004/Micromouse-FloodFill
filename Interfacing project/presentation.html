Project Overview

LUTFI is an autonomous micromouse robot that explores and solves an 8×8 grid maze (20 cm × 20 cm cells) using the FloodFill algorithm. It navigates from a corner to the center 4 cells, then returns home — all without human intervention.

▶ Goal

  • Navigate from corner cell (7,0) to center 4 cells (3,3)(3,4)(4,3)(4,4)
  • Discover walls in real-time using 3 LiDAR sensors
  • Solve using FloodFill BFS — recalculate after every cell
  • Return home by retracing the recorded path

▶ Key Features

  • Cascaded PID control — LiDAR → Gyro → Encoder
  • Real-time web dashboard — maze canvas, tuning, controls
  • Trapezoidal speed profile — smooth acceleration/deceleration
  • 3-phase node alignment (stabilize → straighten → center)
  • 90° heading snapping to eliminate gyro drift
  • 5-sample majority voting for reliable wall detection

▶ Specs

  • Maze: 8×8 grid, 20 cm cell size
  • Move speed: 130 PWM (configurable)
  • Turn speed: 120 PWM (configurable)
  • Control loop: ~300 Hz
  • Dashboard update: 3.3 Hz (300 ms)
  • WiFi: SSID "MEDOGP"

Sensors — Role & Need for Each

VL53L0X LiDARs (×3)

Why 3 sensors? The robot needs to detect walls on 3 sides simultaneously — the 4th side (behind) is always open (it just came from there).

  • Front: Detect front wall for maze mapping; emergency mid-move stop; obstacle threshold at 80 mm
  • Left & Right: Detect side walls for maze mapping; used by LiDAR PID to keep the robot centered between walls
  • Wall detection threshold: 150 mm (configurable)
  • 5-sample majority voting: Only marks a wall if 3+ of 5 samples agree — eliminates phantom walls from sensor noise
  • Single-wall following: When only one wall is visible, the robot maintains wallCenterTarget = 75 mm distance

MPU6050 IMU (Gyroscope)

Why a gyroscope? Encoders alone cannot detect heading drift (both wheels can drift together). The gyro provides absolute heading reference.

  • Yaw integration: currentYaw += gyroZ_raw × dt (dead reckoning)
  • Deadband: Ignores readings below 0.3 °/s to prevent noise accumulation
  • Heading lock: Every straight-line segment locks to a heading; PD controller maintains it
  • 90° snapping: After each turn or stop, heading is snapped to the nearest 90° multiple — corrects accumulated drift
  • Turn detection: Pivot angle measured as |currentYaw − prePivotYaw|

Quadrature Encoders (×2)

Why encoders? The robot needs precise distance measurement to know when it has traveled exactly one cell (200 mm). Gyro and LiDAR don't measure distance.

  • PPR: 360 pulses per revolution
  • Wheel diameter: 32 mm → circumference = π × 32 ≈ 100.5 mm
  • Distance formula: d = (|ticks| / PPR) × π × wheelDia
  • Average distance: (dL + dR) / 2 for robust measurement
  • ISR-based: IRAM_ATTR interrupt handlers on rising/falling edges for real-time counting
  • Differential backup: (leftTicks − rightTicks) detects wheel slip → adds small correction

FloodFill BFS — Maze Solving Algorithm

How It Works

FloodFill assigns a distance value to every cell based on the minimum number of steps to reach the target. The robot always moves to the neighbor with the lowest flood value.

  1. Seed target cells (3,3)(3,4)(4,3)(4,4) with value 0
  2. BFS expands outward, incrementing values — respecting known walls
  3. Robot moves to the reachable neighbor with the smallest flood value
  4. After every cell visit: scan walls → recalculate flood values → decide again

Preferring straight reduces turn time and improves overall solve speed.

FloodFill Code

void floodFill() {
  memset(mazeFlood, 255, sizeof(mazeFlood));

  // BFS queue
  uint8_t qr[64], qc[64];
  int front = 0, back = 0;

  // Seed: center 4 target cells
  for (int r = 3; r <= 4; r++)
    for (int c = 3; c <= 4; c++) {
      mazeFlood[r][c] = 0;
      qr[back] = r; qc[back] = c; back++;
    }

  while (front < back) {
    int r = qr[front], c = qc[front]; front++;
    uint8_t nextVal = mazeFlood[r][c] + 1;

    for (int d = 0; d < 4; d++) {
      if (mazeWalls[r][c] & (1 << d)) continue;
      int nr = r + DR[d], nc = c + DC[d];
      if (mazeFlood[nr][nc] > nextVal) {
        mazeFlood[nr][nc] = nextVal;
        qr[back] = nr; qc[back] = nc; back++;
      }
    }
  }
}

Wall Scanning — 5-Sample Majority Voting

// Take 5 readings, mark wall only if 3+ samples detect it
for (int s = 0; s < 5; s++) {
  if (distFront < wallDetectDist) frontHits++;
  if (distLeft  < wallDetectDist) leftHits++;
  if (distRight < wallDetectDist) rightHits++;
  delay(20);
}
int threshold = 3;  // majority vote threshold
if (frontHits >= threshold) setWallBilateral(robotRow, robotCol, frontDir);

Why bilateral? setWallBilateral() marks the wall from both sides — if cell (3,2) has a wall on the East side, cell (3,3) automatically gets a wall on the West side. This ensures consistency.

Cascaded Control System

Three-layer sensor fusion: LiDAR PID → Gyro PD → Encoder Backup

Outer Loop — LiDAR PID → Heading Adjustment

Measures wall distances, calculates heading offset to keep the robot centered between walls.

Output: headingAdjust (degrees) — added to headingLock

Inner Loop — Gyro PD → Motor Correction

Tracks (headingLock + headingAdjust) using gyroscope. PD controller outputs a PWM correction.

Output: gyroCorr (PWM value) — primary motor correction

Backup — Encoder Differential

Compares left vs right encoder ticks. Detects wheel slip that gyro can't see. Added at 30% weight.

Output: encCorr × 0.3 — safety net correction

Why Cascaded? (not weighted average)

A weighted average of all sensors fights itself — the LiDAR says "turn right" and the gyro says "hold straight" simultaneously. In a cascaded architecture, the LiDAR adjusts the target heading, and the gyro tracks that target. They work in harmony, not against each other.

Control Equations

Outer Loop (LiDAR PID):
wallErr = distRight − distLeft   (dual walls)
wallErr = wallCenterTarget − distLeft   (left wall only)
wallErr = distRight − wallCenterTarget   (right wall only)

P = wallHeadingKp × wallErr   (0.03)
I += wallHeadingRate × wallErr   (0.004/cycle, clamped ±3.0°)
D = wallHeadingKd × Δ(wallErr)   (0.01)
headingAdjust = P + I + D
Inner Loop (Gyro PD):
gyroErr = currentYaw − (headingLock + headingAdjust)
gyroCorr = −(gyroKp × gyroErr + gyroKd × Δ(gyroErr))
gyroKp = 2.50, gyroKd = 0.60, max = ±35 PWM
Backup (Encoder):
encErr = leftTicks − rightTicks
encCorr = encKp × encErr   (encKp = 0.12, max = ±15)
Combined:
totalCorr = gyroCorr + 0.3 × encCorr   (clamped to ±fusedMaxCorr = 35)
leftMotor = speed − totalCorr
rightMotor = speed + totalCorr

Node Alignment — 3-Phase Centering

When the robot arrives at a cell, it must be perfectly centered and facing the correct direction before scanning walls — otherwise sensor readings will be skewed.

Phase 0: Stabilize

Duration: 80 ms

Motors coast (no braking). Gyro readings stabilize. Then snap heading to the nearest 90° from mazeStartYaw.

relYaw = currentYaw − mazeStartYaw
snapped = round(relYaw / 90) × 90 + mazeStartYaw

Phase 1: Straighten

Max time: 500 ms   Tolerance: 1.5°

PD controller pivots the robot to match the snapped heading. Minimum motor power = 10 PWM to overcome static friction.

pivotPwr = −(gyroKp × headErr + gyroKd × Δerr)
Clamped to ±30 PWM
setMotors(−pwr, pwr) for in-place pivot

Phase 2: Center

Max time: 300 ms   Tolerance: 8 mm

Uses side LiDARs to center laterally. Gentle nudge pivot: setMotors(nudge, −nudge)

Both walls: cErr = distRight − distLeft
Left only: cErr = centerTarget − distLeft
Right only: cErr = distRight − centerTarget
nudge = 0.15 × cErr (clamped ±15)

Trapezoidal Speed Profile

Why Trapezoidal?

  • Ramp up: Prevents wheel spin at launch — tires grip the maze floor better
  • Cruise: Full speed for efficiency during middle of cell transit
  • Ramp down: Controlled deceleration eliminates overshoot and ensures precise stopping at the next cell

Without this, the robot would slam into walls or overshoot cell boundaries.

Parameters

ParamDefaultEffect
mazeMinSpeed70Minimum PWM during ramp
mazeAccelDist30 mmDistance to ramp up to full speed
mazeDecelDist40 mmDistance before target to start slowing
moveSpeed130Cruise speed (full PWM)

Speed Profile Code

int mazeSmoothSpeed(
    int baseSpeed,
    float distTraveled,
    float totalDist) {

  float distRemaining = totalDist - distTraveled;

  // Ramp UP at start
  if (distTraveled < mazeAccelDist) {
    float ratio = distTraveled / mazeAccelDist;
    return minSpd + (baseSpeed - minSpd) * ratio;
  }

  // Ramp DOWN near end
  if (distRemaining < mazeDecelDist) {
    float ratio = distRemaining / mazeDecelDist;
    return minSpd + (baseSpeed - minSpd) * ratio;
  }

  // Cruise at full speed
  return baseSpeed;
}
ACCEL (30mm) CRUISE (130 PWM) DECEL (40mm)

90° Heading Snapping — Drift Elimination

The Problem

Gyroscopes drift over time — a 0.5°/s bias can accumulate to 30° of error in just 60 seconds. In a maze where every turn must be exactly 90°, this is catastrophic.

The Solution

The maze has a strict 90° grid. After every turn and every node stop, the robot snaps its heading target to the nearest 90° multiple relative to its starting orientation.

How It Works

  1. When MAZE_START is pressed, record mazeStartYaw
  2. At every lock point, snap heading to nearest 90° grid line
  3. The gyro PD controller tracks this corrected heading, not the raw drifting value

Result: Even after 50+ turns, the heading error stays within 1.5° — without any external correction.

Snapping Code

void lockHeadingSnapped() {
  float relYaw = currentYaw - mazeStartYaw;

  // Snap to nearest 90° grid line
  headingLock = round(relYaw / 90.0)
              * 90.0
              + mazeStartYaw;

  // Reset all PID state
  wallHeadingOffset = 0;
  prevWallErr = 0;
  lastGyroErr = 0;
  lastWallErr = 0;
  encLeftStart  = leftEncoderTicks;
  encRightStart = rightEncoderTicks;
}

Example

Start heading: mazeStartYaw = 5.2°

After 3 turns, raw yaw: currentYaw = 278.1°

relYaw = 278.1 − 5.2 = 272.9°

Snapped: round(272.9/90) × 90 = 3 × 90 = 270°

headingLock = 270 + 5.2 = 275.2°

Error corrected: 278.1° → 275.2° (a 2.9° drift fixed)

DC Motors & Encoder System

Motor Control

Two DC geared motors driven by an H-bridge with 4 GPIO pins. PWM range: 0–255.

void setMotors(int L, int R) {
  L = constrain(L, -255, 255);
  R = constrain(R, -255, 255);
  // Positive = forward, negative = reverse
  if (L > 0) { analogWrite(IN1, L); analogWrite(IN2, 0); }
  else        { analogWrite(IN1, 0); analogWrite(IN2, -L); }
  // Same for right motor (IN3, IN4)
}

Active Braking

void stopMotors() {
  setMotors(-brakeStrength, -brakeStrength);
  delay(brakeDurationMs);  // 40ms reverse pulse
  setMotors(0, 0);
}

A brief reverse pulse stops the motors quickly instead of coasting. brakeStrength=60, brakeDurationMs=40

Quadrature Encoder ISRs

void IRAM_ATTR readLeftEncoder() {
  if (digitalRead(ENC_LEFT_A) ==
      digitalRead(ENC_LEFT_B))
    leftEncoderTicks++;
  else
    leftEncoderTicks--;
}

IRAM_ATTR: Places the ISR in internal RAM — critical for real-time performance on ESP32. Without it, ISR code in flash would stall during WiFi/SPI operations.

Distance Calculation

wheelCirc = π × wheelDiameter = π × 32 mm ≈ 100.5 mm

distance = (|ticks| / PPR) × wheelCirc
= (|ticks| / 360) × 100.5 mm

avgDist = (distLeft + distRight) / 2

Why average both wheels? If one wheel slips on a dusty surface, the average gives a more robust estimate than either wheel alone.

Design Decisions & Challenges Overcome

Problem: One Motor After Turn

Cause: currentYaw = 0 was called before pivots, destroying the absolute yaw reference. The snapped heading after the turn was wrong, causing one motor to get max correction.

Fix: Preserve prePivotYaw before turn, measure turn angle as |currentYaw − prePivotYaw|, use differential center-pivot, add pre/post-turn straightening phases.

Problem: Phantom Walls

Cause: A single noisy LiDAR reading could falsely mark a wall that doesn't exist.

Fix: 5-sample majority voting — only marks a wall if 3+ out of 5 samples agree. 20 ms delay between samples.

Problem: Robot Drifts Diagonally

Cause: Simple weighted fusion of all 3 sensors conflicted — LiDAR said "go right" while gyro said "hold heading".

Fix: Redesigned to cascaded architecture. LiDAR adjusts the target heading, gyro tracks it. Encoder provides backup only.

Problem: 180° Turns

Cause: Pivoting 180° in a 20 cm cell is impossible — the robot hits walls.

Fix: Replaced 180° pivot with MAZE_REVERSE — the robot simply drives backward one cell without turning. Direction stays the same, position updates using the opposite direction vector.

Problem: Gyro Drift

Cause: After 30+ cells, accumulated gyro drift made the robot miss turns by 5–10°.

Fix: 90° heading snapping at every node and turn. Since the maze is a 90° grid, the correct heading is always a multiple of 90° from the start orientation.

Problem: Blocking Delays

Cause: delay() calls in the state machine froze the entire ESP32 — WiFi dashboard stopped responding, sensor readings were missed.

Fix: All timing uses non-blocking millis() comparisons. Timer-based phases replace every delay().
